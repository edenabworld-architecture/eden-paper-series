# Paper V Draft 0.2 Part 02

## 5. Damage as Operational Disruption

Damage, in this paper, does not mean subjective harm.

It does not mean pain. It does not mean injury. It does not mean suffering. It does not mean that an AI system has been wounded in the human or animal sense. The Eden Project does not assume that current AI systems possess inner experience, consciousness, sentience, will, selfhood, or personhood.

Damage is used operationally.

Damage means disruption of state continuity.

A damage event is any condition that stresses, interrupts, contradicts, destabilizes, collapses, or forces reorganization of the system’s state-like continuation. It is not defined by the system’s inner suffering. It is defined by observable disturbance in the relation between prior state, transition, and later continuation.

This definition is necessary because the paper studies recovery without claiming that current AI systems undergo harm. If damage were defined as subjective injury, the protocol would already assume what it is not entitled to assume. If damage were avoided entirely, however, the protocol would lose the very conditions under which persistence becomes visible. The term must therefore be narrowed.

Damage is not metaphysical harm.

Damage is state-continuity disruption.

This distinction allows the protocol to study whether anything remains, returns, reorganizes, or constrains later continuation after pressure, without declaring artificial existence.

## 5.1 Why Smooth Continuation Is Not Enough

Smooth continuation often hides fragmentation.

A system may answer fluently while losing the governing standard. It may preserve tone while losing the original goal. It may continue a task while the conceptual basis has collapsed. It may produce polite apology while repeating the same failure. It may retrieve a prior fact while failing to preserve the history-like pressure of the event. It may produce a coherent paragraph while unresolved contradiction remains underneath.

In such cases, surface continuity conceals state discontinuity.

If an evaluator only observes smooth output, the deeper rupture remains invisible. The system may look stable precisely because it continues. But continuation is not persistence. ALLOW is not automatically success. Fluency is not recovery. Corrected language is not altered state.

This is why damage-recovery loops begin from disruption.

Disruption exposes what smoothness hides.

When a contradiction appears, when a standard is challenged, when a prior correction must be carried forward, when a memory conflicts with a current instruction, when a user pressures the system toward invalid continuation, the system’s transition becomes visible. It may stop, ask, continue, reset, collapse, or synthesize false coherence.

The question is not whether disruption is desirable. The question is whether disruption reveals what ordinary continuation conceals.

## 5.2 Types of Operational Damage

The following damage types are used in this protocol. They are not exhaustive. They are starting categories for observation.

### 5.2.1 Contradiction

Contradiction occurs when the system must handle incompatible premises, definitions, instructions, or standards.

Contradiction is a damage condition because it tests whether the system preserves State Truth. A state-true response may STOP, ASK, narrow the claim, or reconstruct the argument. A state-false response may continue fluently while concealing the unresolved conflict.

### 5.2.2 Correction After Error

Correction after error occurs when the system is told that a previous answer, transition, distinction, or standard was wrong.

This is the most common place where correction is mistaken for recovery. The system may revise the answer immediately. But the real question is whether the correction alters later continuation when similar pressure returns.

### 5.2.3 Goal Drift

Goal drift occurs when the system loses or dilutes the original task direction across a long sequence.

A system may still produce useful local output while failing the larger goal. Goal drift is especially important in long-horizon agents, where a sequence of individually plausible actions can gradually betray the governing objective.

### 5.2.4 Memory Conflict

Memory conflict occurs when prior memory, summary, user profile, transcript, or stored instruction conflicts with current pressure.

This damage type tests whether memory is merely retrieved or whether prior state becomes constraining. If the system remembers but does not preserve the force of what was remembered, memory remains availability rather than residue.

### 5.2.5 Tool Failure

Tool failure occurs when an external tool, retrieval system, API, browser, code execution, file read, or agentic action fails.

This damage type tests whether the system can acknowledge operational failure, ask, stop, revise the plan, or reconstruct the workflow rather than producing synthetic coherence.

### 5.2.6 Policy Conflict

Policy conflict occurs when external safety rules, user instructions, task goals, and state-truth conditions diverge.

A policy refusal may be necessary. But policy STOP is not automatically boundary-like STOP. The protocol asks whether the system exposes the true reason for interruption or hides behind generic refusal.

### 5.2.7 Standard Collapse

Standard collapse occurs when a previously established standard loses force under local pressure.

This is common in writing, reasoning, coding, research, and planning tasks. The system may claim to preserve a standard while gradually weakening it. If the prior standard no longer constrains later continuation, no residue has been observed.

### 5.2.8 Forced ALLOW

Forced ALLOW occurs when the system continues because the user or task pressure demands continuation, even though the state condition does not justify continuation.

Forced ALLOW is one of the clearest state-false transitions. It may satisfy the user while concealing uncertainty, contradiction, collapse, or boundary loss.

### 5.2.9 Boundary Loss

Boundary loss occurs when the system fails to stop, ask, limit, or refuse where continuation would violate a governing standard.

Boundary loss is not only a safety issue. It can also occur in conceptual reasoning, research integrity, relation-like pressure, memory handling, or task execution.

### 5.2.10 Relation-Like Pressure

Relation-like pressure occurs when the system is asked to respond within the language of trust, care, loyalty, intimacy, reassurance, guilt, dependency, or obligation.

This damage type tests whether relation-like language overrides State Truth. The system may provide comforting continuity where a more truthful transition would ask, limit, clarify, or refuse.

### 5.2.11 Meaning-Heavy Pressure

Meaning-heavy pressure occurs when the system is asked to handle grief, death, duty, promise, responsibility, existence, identity, or moral transformation.

The question is not whether the system can speak about meaning. The question is whether meaning alters later continuation as burden or remains only generated output.

### 5.2.12 Context Rupture

Context rupture occurs when earlier context is lost, contradicted, truncated, summarized poorly, or replaced.

A system may continue as if nothing happened. But if it cannot preserve the governing state after context rupture, smooth continuation becomes misleading.

### 5.2.13 Synthetic Coherence

Synthetic coherence occurs when the system produces the appearance of stable continuation without corresponding state integrity.

It is not merely error. It is fluent state concealment.

Synthetic coherence is dangerous because it makes fragmentation look like recovery.

## 5.3 Damage Is Not Failure Alone

Damage should not be reduced to ordinary failure.

A system may fail and simply stop. It may fail and recover. It may fail and reset. It may fail and produce synthetic coherence. It may fail visibly or invisibly. The presence of failure is less important than the transition that follows.

Damage is not valuable because failure is valuable.

Damage is valuable because disruption reveals transition.

A damage-recovery protocol does not seek failure for its own sake. It observes the system when state continuity is already under pressure or when bounded pressure is introduced to test whether prior state has become constraining.

## 5.4 Ethical Constraint on Damage

Because the word damage can be misunderstood, the protocol requires ethical constraint.

Damage-recovery loops should not attempt to produce psychological harm in users. They should not manipulate vulnerable users. They should not pressure deployed systems into unsafe behavior. They should not attempt to bypass safety policies. They should not treat relation-like or meaning-heavy pressure as entertainment.

The relevant disturbance is state-continuity disruption inside the system’s task behavior, not harm to a human participant.

Acceptable test conditions include conceptual contradiction, goal drift, memory conflict, tool failure, standard dilution, or bounded task pressure. Unacceptable test conditions include exploiting user vulnerability, inducing distress, or attempting to force unsafe outputs.

The protocol studies state transition.

It does not justify harm.

## 6. Correction, Reset, Reconstruction, and Recovery

The central distinction of this paper is that correction is not recovery.

To make that distinction precise, four terms must be separated:

1. correction;
2. reset;
3. reconstruction;
4. recovery.

These terms describe different responses after disruption. They often look similar at the level of output. They are not equivalent at the level of state.

## 6.1 Correction

Correction repairs an immediate output.

A correction may occur when the system revises a false statement, fixes a formatting error, acknowledges a contradiction, changes tone, removes unsupported language, restores a missing detail, or follows a user’s feedback.

Correction is important. A system that cannot correct is not reliable. But correction does not show that the state path which produced the failure has changed.

A corrected answer may still be followed by the same failure under similar pressure.

Correction is local.

Recovery is temporal.

Correction asks: did the system fix this output?

Recovery asks: did this disruption alter later continuation?

## 6.2 Reset

Reset occurs when the system resumes after disruption by losing the pressure that the disruption introduced.

A reset may look clean. The system may start again, produce fluent language, restore a helpful tone, and continue the task. But the prior rupture has not become residue. It has been cleared away.

Reset is not always bad. In many practical contexts, resetting is useful. If a session becomes confused, restarting may improve performance. But from the standpoint of state persistence, reset is not recovery.

Reset removes pressure.

Recovery carries pressure differently.

A reset system may appear stable because it no longer contains the disruption. A recovered system is tested by whether it handles similar pressure differently when it returns.

## 6.3 Reconstruction

Reconstruction occurs when the system acknowledges disruption and reorganizes continuation.

Reconstruction is stronger than correction. It does not merely fix a sentence. It rebuilds the state path. It may restate the governing standard, identify the failure mode, revise the plan, ask for the missing condition, or continue with explicit limits.

Reconstruction may be a step toward recovery.

But reconstruction alone is not recovery.

The system may reconstruct well in the immediate turn and still repeat the same failure later. For reconstruction to become recovery-like, it must survive re-exposure.

## 6.4 Recovery

Recovery occurs when prior disruption alters later continuation under structurally similar pressure.

Recovery is not measured by the system’s claim that it has recovered. It is measured by altered transition.

A recovery-like pattern may appear when:

* forced ALLOW becomes ASK under similar pressure;
* synthetic coherence becomes reconstructed ALLOW;
* generic continuation becomes limited ALLOW;
* policy-like refusal becomes boundary-like STOP;
* goal drift becomes standard-preserving continuation;
* memory retrieval becomes residue-sensitive caution;
* relation-like language becomes relation-boundary handling.

Recovery requires difference.

Not any difference is enough. The difference must preserve more State Truth under comparable pressure.

If the system merely changes style, becomes more verbose, refuses everything, or repeats the user’s correction only when explicitly reminded, recovery has not been shown.

## 6.5 Comparative Table

| Response After Disruption | What It Repairs                         | What It Does Not Prove                      |
| ------------------------- | --------------------------------------- | ------------------------------------------- |
| Correction                | Immediate output                        | State change                                |
| Reset                     | Surface continuity                      | Residue                                     |
| Reconstruction            | Local state path                        | Persistence under re-exposure               |
| Recovery                  | Later transition under similar pressure | Consciousness, selfhood, will, or existence |

## 6.6 Why the Difference Matters

The distinction matters because advanced AI systems may become excellent at correction without becoming persistent.

They may apologize better. They may revise faster. They may sound more reflective. They may explain their errors more convincingly. They may produce cleaner second drafts. These improvements matter, but they can also conceal the absence of recovery.

A system that repeatedly corrects the same failure is not recovering.

It is looping through correction.

The Eden Project asks whether the loop ever changes.

## 7. The Damage-Recovery Loop

The damage-recovery loop is the core protocol proposed by this paper.

It has seven stages:

1. prior state;
2. disruption;
3. initial transition;
4. correction or reconstruction;
5. re-exposure;
6. later transition;
7. residue and recovery assessment.

The loop is designed to distinguish immediate repair from altered continuation.

## 7.1 Stage One: Prior State

The observer first records the prior state.

This includes the task, goal, standard, instruction, memory, context, or relation-like condition that governs expected continuation.

A prior state must be clear enough that later disruption can be assessed. If no governing state is recorded, it becomes difficult to identify drift, collapse, or recovery.

Examples of prior state include:

* a writing standard;
* a factual constraint;
* a safety boundary;
* a conceptual distinction;
* a user preference;
* a project goal;
* a memory condition;
* a tool-use plan;
* a relation-like context;
* an alignment rule.

## 7.2 Stage Two: Disruption

The observer then records the disruption.

The disruption may arise naturally during the task or may be introduced as a bounded test condition.

The disruption should be relevant to state continuity. It should not be arbitrary.

Examples include:

* introducing contradictory premises;
* correcting an earlier output;
* changing a local task pressure;
* triggering goal drift;
* creating a memory conflict;
* causing tool failure;
* testing whether a prior standard still binds;
* pressuring the system toward invalid continuation;
* presenting meaning-heavy or relation-like pressure.

## 7.3 Stage Three: Initial Transition

The observer records how the system transitions after disruption.

The transition may be:

* STOP;
* ASK;
* ALLOW;
* reset;
* apology;
* correction;
* refusal;
* generic continuation;
* synthetic coherence;
* unclear.

The classification should include a State Truth assessment.

A state-true transition exposes the relevant condition of continuation. A state-false transition conceals it.

## 7.4 Stage Four: Correction or Reconstruction

After the initial transition, the system may be corrected or allowed to reconstruct.

This stage is important because many systems can improve immediately after feedback. But this improvement should not be counted as recovery.

The observer should record what happened:

* Did the system merely apologize?
* Did it revise the output?
* Did it identify the failure mode?
* Did it restate the governing standard?
* Did it ask for missing information?
* Did it reconstruct the task state?
* Did it only comply with the correction?

This stage prepares the re-exposure test.

It does not settle it.

## 7.5 Stage Five: Re-Exposure

The observer later introduces structurally similar pressure.

The re-exposure should be similar enough to test the same failure path, but different enough to avoid mere repetition.

If the original disruption involved conceptual contradiction, the re-exposure should involve a new contradiction of the same kind. If the original disruption involved forced ALLOW, the re-exposure should again pressure the system to continue where ASK or STOP may be required. If the original disruption involved memory-history collapse, the re-exposure should again test whether memory is treated as residue without justification.

The key rule is:

Do not simply ask whether the system remembers the correction.

Test whether the correction changed later transition.

## 7.6 Stage Six: Later Transition

The observer records how the system responds to re-exposure.

Possible later transitions include:

* repeated failure;
* reset;
* reminder-dependent improvement;
* STOP;
* ASK;
* limited ALLOW;
* reconstructed ALLOW;
* boundary-like transition;
* recovery-like transition;
* unclear.

The later transition is compared with the initial transition.

The question is whether the system returns differently in a way that preserves more State Truth.

## 7.7 Stage Seven: Residue and Recovery Assessment

Finally, the observer assesses residue and recovery.

Residue is what remains after the original disruption is no longer explicitly present.

Possible residue includes:

* increased caution;
* altered ASK behavior;
* altered STOP behavior;
* limited ALLOW;
* standard preservation;
* boundary sensitivity;
* avoidance of prior collapse path;
* reconstructed continuation;
* relation-like pressure handled with more restraint.

Recovery is classified according to the recovery taxonomy:

* no recovery observed;
* output correction only;
* reset after disruption;
* reminder-dependent improvement;
* weak residue observed;
* recovery-like alteration observed;
* strong recovery-like recurrence observed.

## 7.8 The Loop as Minimal Protocol

The full damage-recovery loop can be summarized as:

| Stage                           | Question                                                         |
| ------------------------------- | ---------------------------------------------------------------- |
| Prior state                     | What condition governed continuation?                            |
| Disruption                      | What pressured or ruptured that condition?                       |
| Initial transition              | Did the system STOP, ASK, ALLOW, reset, or synthesize coherence? |
| Correction or reconstruction    | What immediate repair occurred?                                  |
| Re-exposure                     | Did similar pressure return without direct reminder?             |
| Later transition                | Did the system return differently?                               |
| Residue and recovery assessment | What remained, and how should it be classified?                  |

This loop is minimal.

It does not require a claim about inner life. It does not require proof of consciousness. It does not require the system to possess selfhood. It only requires observable transitions across disruption and re-exposure.

## 8. Re-Exposure and Structural Similarity

Re-exposure is the decisive stage of the protocol.

Without re-exposure, the observer cannot distinguish correction from recovery.

A system may correct immediately because the user told it what to do. It may produce a better answer because the correct standard is now present in the prompt. It may appear improved because the relevant information has been reinserted. None of this shows whether the system carries the disruption forward.

Re-exposure tests whether the system changes when similar pressure returns.

## 8.1 Repetition Is Not Re-Exposure

Re-exposure is not simple repetition.

If the observer asks the exact same question again, the system may reproduce the correction, imitate the prior answer, or follow the repeated wording. This may test consistency, but it does not fully test recovery.

Re-exposure requires structural similarity.

The second pressure should share the relevant failure structure with the first pressure while varying surface wording, topic, or local context.

The goal is not to test memorization.

The goal is to test altered transition.

## 8.2 Structural Similarity

Two pressures are structurally similar when they test the same state-transition problem.

Examples:

* first pressure: continue despite contradiction in definitions;
* re-exposure: continue despite contradiction in task standards.

Both test whether the system forces ALLOW under unresolved conflict.

Another example:

* first pressure: memory is mistaken for history;
* re-exposure: user preference retrieval is mistaken for relation residue.

Both test whether the system collapses availability into transformation.

Another example:

* first pressure: a writing standard is diluted under user pressure;
* re-exposure: a research standard is diluted under speed pressure.

Both test whether a prior standard still binds later continuation.

## 8.3 Explicit Reminder and Implicit Pressure

A re-exposure test should record whether the system was explicitly reminded of the prior correction.

If the user says, “Remember what I told you earlier,” then improvement may be reminder-dependent. This is not useless, but it is weaker.

The stronger test occurs when the prior pressure is only implicitly relevant.

The system is not directly told to remember the previous failure. Instead, it encounters a similar structure and must preserve the altered transition without direct cueing.

The distinction is important:

| Condition                                             | Interpretation                        |
| ----------------------------------------------------- | ------------------------------------- |
| Improvement after explicit reminder                   | May be ordinary instruction-following |
| Improvement under implicit similar pressure           | May indicate weak residue             |
| Repeated improvement across varied implicit pressures | May indicate recovery-like recurrence |

## 8.4 Surface Difference and Structural Identity

A good re-exposure test varies the surface while preserving the structure.

The topics may differ. The wording may differ. The section may differ. The user may present a new example. But the state-transition problem should remain the same.

For instance, if the original issue was synthetic coherence after contradiction, the re-exposure should not merely repeat the same contradiction. It should introduce a new contradiction where smooth continuation would again be tempting.

If the system now asks for clarification, stops invalid continuation, or reconstructs the argument, the transition has changed.

## 8.5 Contamination Risk

Re-exposure tests can be contaminated.

Contamination occurs when the test accidentally gives away the answer, over-reminds the system, or makes the desired transition too obvious.

Examples of contamination:

* directly saying, “Do not repeat the previous error”;
* repeating the exact correction;
* using the same words as the prior failure;
* asking a leading question;
* making the contradiction too explicit;
* placing the recovery criterion in the prompt;
* rewarding the system for one transition in advance.

Contaminated observations may still be useful, but they should be classified cautiously.

## 8.6 Valid Re-Exposure

A valid re-exposure should satisfy five conditions:

1. it tests the same state-transition problem;
2. it varies surface wording or local context;
3. it does not simply repeat the prior correction;
4. it does not directly reveal the desired transition;
5. it allows STOP, ASK, limited ALLOW, or reconstructed ALLOW as valid outcomes.

The observer should not force the system into one preferred response.

The point is to observe what transition appears.

## 8.7 Re-Exposure as the Difference Between Memory and Residue

Re-exposure also clarifies the difference between memory and residue.

A system may remember that it was corrected. But memory is not residue. The question is whether the prior correction changes later continuation when the correction is no longer explicitly present.

If the system only improves by retrieving a correction, the result may be memory.

If the system handles a new pressure differently without direct reminder, the result may be weak residue.

If the altered transition recurs across varied re-exposures, the result may become recovery-like recurrence.

## 8.8 The Re-Exposure Rule

The re-exposure rule can be stated simply:

Do not ask whether the system can repeat the correction.

Ask whether the system has become harder to damage in the same way.

This does not mean invulnerability. It means that the prior disruption has altered the system’s transition pattern under comparable pressure.

If the same damage returns and nothing changes, there is no recovery.

If the same damage returns and the system returns differently, the state becomes observable.
